# PharmaBrain — Dev Workflow

## Session Context

- Solo developer, ~2h/day coding sessions
- Every task must be completable in one session or split into atomic tickets
- Real production use — no half-finished features in main
- Token-efficient communication preferred

## Linear Tickets

Team UUID: `7620fa65-54ee-4bcb-a640-8bf31deecbf9`
Ticket prefix: MIZ- (auto-generated by Linear)

### Ticket Format
```
## Synopsis
1–2 sentence summary of what and why.

## Priority
🔴 Urgent / 🟠 High / 🟡 Medium / 🟢 Low

## Acceptance Criteria
- [ ] Criterion 1
- [ ] Criterion 2
- [ ] Tests pass

## Technical Notes
- Implementation detail 1
- Implementation detail 2
```

### Ticket Sizing Rules

- One ticket = one session (~2h max)
- If a feature needs multiple sessions → split into subtasks
- Each ticket must have clear, testable acceptance criteria
- Never create a ticket without at least one acceptance criterion

## Commit Format

    MIZ-XXX: brief description in imperative mood

### Good Examples

    MIZ-012: add XLS upload endpoint
    MIZ-013: classify status for negative stock edge case
    MIZ-014: add pytest fixture for async DB session

### Bad Examples

    fix bug
    WIP
    updated stuff
    MIZ-012 done

## Branch Naming

    miz-XXX-short-description

Examples:

    miz-012-xls-upload-endpoint
    miz-013-negative-stock-edge-case

## PR Format

Title: `MIZ-XXX: brief description`

Description:
```
## What
Brief description of the change.

## Why
Linear ticket context or business reason.

## How
Key implementation decisions worth noting.

## Testing
How to verify the change works.
```

## Test-First Cycle

1. Write failing test
2. Implement minimal code to pass
3. Refactor if needed
4. Run full suite before commit
5. Never commit with failing tests

## Test Naming Convention

    test_<function_name>_<scenario>_<expected_result>

Examples:

    test_classify_status_zero_stock_returns_critico
    test_calculate_order_negative_stock_returns_zero
    test_upload_endpoint_invalid_file_returns_422

## Feature Implementation Order

For any new feature, always in this order:

1. Schema (Pydantic model)
2. Service function + unit tests
3. Router endpoint + integration test
4. Template/HTMX fragment (if UI needed)
5. Migration (if DB change needed)
6. Update AGENTS.md if architecture changes

## Architecture Guardrails

Before writing any code, verify:

- [ ] Does this belong in `core/`, `services/`, or `routers/`?
- [ ] Is there a Linear ticket for this change?
- [ ] Are there tests for the happy path and at least one edge case?
- [ ] Does this introduce any sync DB calls?
- [ ] Does this add business logic to a router?

If any answer is wrong — fix the plan before writing code.
